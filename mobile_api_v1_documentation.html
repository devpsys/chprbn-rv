Overview

All endpoints below share this base path (no trailing slash):

https://<your-host>/api/v1/mobile

Example production host: chprbn.gov.ng → https://chprbn.gov.ng/api/v1/mobile/…

Login eligibility: Active practitioners with tutor access (is_tutor) may obtain a token. Credentials use license number as username and the account password.

Request & response conventions

Authenticated requests

After login, send the personal access token on every protected call:

Authorization: Bearer <token> Content-Type: application/json Accept: application/json

Successful responses (envelope)

Most success payloads use:

{ "status": true, "message": "…", "data": { … } }

The data object shape depends on the endpoint (see each section).

Practitioner verified sync

Records a field verification event against an existing practitioner license. The authenticated officer is stored server-side.

POST /practitioners/verified-sync

POST /api/v1/mobile/practitioners/verified-sync

Headers

Authorization: Bearer <token> Content-Type: application/json

Request body (JSON)

FieldTypeRequiredDescription
license_numberstringYesRegistration number (max 100 chars)
verification_locationstringYesWhere verification took place (max 2000 chars)
practitioner_presentbooleanYestrue / false
remarkstringYesFree-text remark (max 5000 chars)
verified_atintegerYesUnix time in milliseconds (UTC)

Example request

{ "license_number": "CH-99201-B", "verification_location": "Lagos University Teaching Hospital — Ward 4", "practitioner_present": true, "remark": "ID matched; license card presented.", "verified_at": 1710858600000 }

Success 200

{ "status": true, "message": "Verification recorded.", "data": { "id": 15, "license_number": "CH-99201-B", "verified_at": 1710858600000 } }

Failure

HTTPWhenBody
422Validation failed{"message":"…"} (first error)
404No practitioner for license_number{"message":"Practitioner not found for this license number."}
403Authenticated user is not a Practitioner model{"message":"Forbidden"}
401Missing / invalid token{"message":"Unauthenticated."}
Note: verified_at is stored in UTC with second precision (milliseconds are truncated when persisted to the database timestamp).

Error responses (all mobile JSON API routes)

Failures use a single key and an appropriate HTTP status:

{ "message": "Human-readable explanation" }

Typical status codes

CodeUse
400Bad request (e.g. missing identifier)
401Unauthenticated / invalid credentials / invalid API key (lookup)
403Forbidden / insufficient permission
404Resource not found
422Validation error (first message only in body)
429Rate limit exceeded (lookup API key)
500Server error

Under /api/*, validation exceptions and unhandled JSON errors follow the same message-only pattern where applicable.
